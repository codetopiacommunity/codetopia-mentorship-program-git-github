# Session 2 — Making a Great PR to a Stranger's Project

> **XP:** 75 (attendance + exercises)

## Overview

Last session you found the issue. This session you ship the PR.

The whole loop: fork, clone, branch, change, commit, push, open PR, respond to feedback. Plus the soft-skill spine that holds it together — writing a PR description that a stranger wants to read, and taking feedback gracefully.

## Objectives

- Run the full fork-and-PR workflow end-to-end.
- Keep your fork in sync with upstream.
- Write a PR description that answers the maintainer's four questions before they ask: *what, why, how, how-tested*.
- Respond to maintainer feedback professionally, including feedback you disagree with.
- Recognize and uphold the unwritten social contract of open source.

## Mastery checks

- [ ] I can set up `origin` (my fork) and `upstream` (source repo) and explain each.
- [ ] I can write a PR description that would pass a maintainer's "do I need to ask more?" filter.
- [ ] I can rebase my PR branch on updated upstream without breaking anything.
- [ ] I can reply to a "please change this" comment in a way that's collaborative, not defensive.
