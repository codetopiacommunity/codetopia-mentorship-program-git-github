# Week 7 Capstone — "Your First Real OSS Contribution"

> **XP:** 150 base + up to 50 stretch

## Brief

This is the capstone where the PR might be real. You will either:

- **Path A (preferred):** Open a **real pull request** against a real open-source project, of any size. Tiny is fine. Doc fixes count. What matters is the loop.
- **Path B (fallback):** If by the end of the week you've not been able to open a real PR (maintainer unresponsive, CLA blocker, nothing ready to ship), complete a **thorough simulated PR** against a chosen target, with a full writeup of what you would have done.

Either path earns full credit. The point is the craft, not the merge.

## Requirements (both paths)

- A `screenshots/` folder in your project containing at least 2 screenshots showing the work in action (e.g., the opened PR page, a maintainer's review comment, your local test passing). See [CONTRIBUTING.md → Screenshot conventions](../../../CONTRIBUTING.md#screenshot-conventions).
- A target project identified, with justification for why it's the right pick.
- A clear, well-scoped change — one logical thing.
- A branch named conventionally.
- A commit (or commits) following the target's style guide.
- A PR description with the four sections: **What / Why / How / Testing**.
- A `WRITEUP.md` covering:
  - How you found the issue.
  - How you evaluated project health.
  - Any interactions with maintainers (quoted, with context).
  - What you learned.

## Path A specifics

- Open the real PR.
- Include its URL in your writeup.
- You do NOT need the PR to be merged by the end of the week. Open-and-waiting is a valid outcome.
- Document any review comments and your responses in the writeup (even if the PR is still in review).

## Path B specifics

- Do everything except `git push` to upstream / opening the PR.
- Include a file `PR-DESCRIPTION.md` with the exact description you *would* have posted.
- Include a `SIMULATED-REVIEW.md` with two review comments you imagine a maintainer making, and your responses.
- State clearly in your writeup *why* you chose the simulated path (e.g., "the issue was claimed by someone else two days after I commented").

## Deliverable

Submit your work as follows:

1. **Open a PR** (the submission of record) to our cohort repo adding your submission under `projects-showcase/week-7-projects/<your-handle>/`. Include:

   - `WRITEUP.md`
   - `PR-DESCRIPTION.md` (both paths — Path A can paste their real description here too)
   - For Path B only: `SIMULATED-REVIEW.md`
   - Your `screenshots/` folder.

2. **In the WhatsApp group** (the celebration): once your PR is open, share your best screenshot and the PR link so the cohort can cheer you on.

## Grading

See [`rubric.md`](./rubric.md). **Pass = 70+.** Use [`checklist.md`](./checklist.md) before submitting.

## Stretch (+50 XP)

- Your PR gets merged before the week is out (+25). (Yes, we reward the universe cooperating — but only because it means you kept the scope tight.)
- You respond to a substantive review comment and ship the requested change (+15).
- You open a *second* PR on the same or a different project (+10).

## Etiquette reminders

Before you type anything into an issue or PR comment:

- Have you read CONTRIBUTING.md?
- Is the issue unclaimed, or did you check with the current claimant?
- Is your question specific enough to answer in <5 minutes?
- Have you thanked anyone yet?

## Emotional preparedness

Your PR might:

- Get merged in 20 minutes.
- Sit for a month with no response.
- Get closed with "not something we want to do."
- Get asked for changes that turn into a weeks-long conversation.

**All of these are success.** You learned the loop.
